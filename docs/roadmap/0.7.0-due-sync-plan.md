# 0.7.0 Due, Time, and Recurring Sync Implementation

This document records the `0.7.0` implementation scope. The goal is to support Todoist due dates with fewer corner cases while keeping Markdown predictable.

## Objective

`0.7.0` should make due handling explicit and structured:

- Keep the Markdown user format simple.
- Preserve Todoist due metadata needed for safe round trips.
- Allow edits only when Sync Todoist can represent the result safely.
- Keep Daily Note generated rows completion-focused and non-destructive.

## API Grounding

Todoist API v1 due objects use a small set of date/time shapes:

| Todoist due shape | Example | Meaning | 0.7.0 handling |
|---|---|---|---|
| All-day date | `2026-06-01` | Date without time | Editable from Markdown. |
| Floating time | `2026-06-01T15:00:00` | Time in the user's Todoist locale, no fixed timezone | Editable as local wall-clock time from Markdown. |
| Fixed time | `2026-06-01T22:00:00Z` plus timezone metadata | Absolute time with timezone semantics | Display as local time, preserve raw metadata, do not silently downgrade. |
| Recurring occurrence | Same date/time shapes plus `is_recurring`, `string`, and `lang` | Current occurrence plus recurrence rule | Completion is supported; rule editing is not supported in Markdown for `0.7.0`. |

References:

- Todoist API v1 due dates: https://developer.todoist.com/api/v1/
- Todoist API v1 tasks create/update fields: https://developer.todoist.com/api/v1/

## Markdown Contract

User-facing Markdown should support only structured values:

| Markdown | Supported? | Behavior |
|---|---:|---|
| `📅 2026-06-01` | Yes | All-day due date. |
| `📅 2026-06-01 15:00` | Yes | Floating/local wall-clock time. |
| `due:2026-06-01` | Yes | Alias for all-day due date. |
| `due:2026-06-01 15:00` | Yes | Alias for floating/local wall-clock time. |
| `every Friday at 15:00` | No | Do not parse as a Todoist recurrence rule. |
| `tomorrow at 5pm` | No | Do not parse as natural-language due input. |

The sync tag and task id remain unchanged:

```markdown
- [ ] Pay card #todo 📅 2026-06-01 15:00 <!-- todoist-id:abc123 -->
```

## Hidden Preservation Metadata

Use hidden metadata only for values that cannot be represented safely in the visible Markdown contract.

```markdown
<!-- todoist-due:{"kind":"recurring","date":"2026-06-01T15:00:00","string":"every month on the 1st at 15:00","lang":"en","timezone":null,"isRecurring":true} -->
```

Rules:

- Metadata is generated by the plugin, not intended for user editing.
- The metadata must be removed/replaced when Todoist changes the due object.
- If metadata is malformed, ignore it and fall back to the Todoist API state during the next sync.
- Never trust metadata over the current Todoist task fetched during sync.

## Internal Model

Replace date-only `dueDate` as the only sync primitive with a structured due model.

| Field | Type | Purpose |
|---|---|---|
| `kind` | `none`, `date`, `floating`, `fixed`, `recurring` | Main safety branch. |
| `date` | `YYYY-MM-DD` | Local date shown in Markdown. |
| `time` | `HH:mm` | Optional local time shown in Markdown. |
| `rawDate` | string | Todoist `due.date`. |
| `timezone` | string or null | Todoist timezone metadata when present. |
| `string` | string or null | Todoist due string for recurrence preservation. |
| `lang` | string or null | Todoist due language. |
| `isRecurring` | boolean | Whether Todoist says the task recurs. |
| `source` | `markdown`, `todoist`, `metadata` | Where the value came from. |

## Sync Rules

| Scenario | Push to Todoist? | Pull to Obsidian? | Reason |
|---|---:|---:|---|
| Ordinary all-day task, date edited in Markdown | Yes | No | Fully representable. |
| Ordinary floating-time task, date/time edited in Markdown | Yes | No | Fully representable as local wall-clock time. |
| Ordinary fixed-time task, visible date/time edited in Markdown | No by default | Yes | Avoid timezone downgrade. |
| Recurring task, visible date/time edited in Markdown | No | Yes | Avoid turning recurrence into one-time due. |
| Recurring task, checkbox checked in ordinary note | Complete occurrence | Then pull next occurrence | Todoist owns recurrence advancement. |
| Recurring task, checkbox checked in Daily Note | Complete only if row date matches current Todoist occurrence | Then rewrite Daily Note | Prevent old rows from completing future occurrences. |
| Query Block checkbox clicked | Direct Todoist complete/reopen | Refresh view | Query Blocks are not Markdown source. |
| Todoist due changed externally | No push from stale Markdown | Yes | Todoist is source of truth for complex due kinds. |

## Implementation Tasks

| Area | Change | Notes |
|---|---|---|
| Types | Add `StructuredDue` and keep legacy `dueDate` only as compatibility surface | Avoid broad string-only comparisons. |
| Parser | Parse `📅 YYYY-MM-DD`, `📅 YYYY-MM-DD HH:mm`, `due:YYYY-MM-DD`, `due:YYYY-MM-DD HH:mm` | Reject natural language by design. |
| Formatter | Render all-day and floating time visibly; attach hidden due metadata for fixed/recurring | Keep output stable and source-mode readable. |
| Todoist service | Normalize API due objects into `StructuredDue` | Preserve `date`, `datetime`, `timezone`, `string`, `lang`, `is_recurring`. |
| Todoist update | Add separate update paths for all-day and floating time; keep recurring/fixed guarded | Use Todoist-supported fields without guessing recurrence rules. |
| Hashing | Include structured due fields that are user-editable; exclude preserved raw metadata unless it affects visible output | Prevent false conflicts. |
| Conflict handling | Treat recurring/fixed due edits from Markdown as non-pushable and pull Todoist due back | Make safe behavior deterministic. |
| Import | Import tasks with visible due and hidden metadata when needed | Imported recurring tasks must not lose rules. |
| Daily Note | Continue generated-row policy: completion only, no due/title/project/label push | Generated content remains non-authoritative. |
| Query Blocks | Format due dates consistently with task formatter | No Markdown metadata needed because this is rendered HTML. |
| README | Document supported Markdown due syntax and unsupported natural language | Avoid user confusion. |

## Test Plan

### Unit Tests

| Test | Expected result |
|---|---|
| Parse `📅 2026-06-01` | `kind=date`, `date=2026-06-01`, no time. |
| Parse `📅 2026-06-01 15:00` | `kind=floating`, `date=2026-06-01`, `time=15:00`. |
| Parse `due:2026-06-01 15:00` | Same structured due as emoji form. |
| Parse `every Friday at 15:00` | Does not become due metadata. |
| Format all-day due | Emits `📅 YYYY-MM-DD`. |
| Format floating due | Emits `📅 YYYY-MM-DD HH:mm`. |
| Format fixed due | Emits visible local time plus hidden preservation metadata. |
| Format recurring due | Emits current occurrence plus hidden preservation metadata. |
| Malformed hidden metadata | Ignored without throwing. |
| Hash all-day date edit | Changes hash. |
| Hash floating time edit | Changes hash. |
| Hidden metadata-only refresh | Does not create a user-edit conflict unless visible due changed. |
| Ordinary all-day date push | Sends Todoist update. |
| Ordinary floating time push | Sends Todoist update with time preserved. |
| Fixed-time Markdown edit | Does not push unsafe due update. |
| Recurring Markdown date edit | Does not push unsafe due update and restores Todoist occurrence. |
| Daily Note checked current recurring row | Completes current occurrence. |
| Daily Note checked old recurring row | Does not complete next occurrence. |

## Release Criteria

- `npm test` covers parser, formatter, hashing, sync rules, and Todoist update payload decisions.
- `npx tsc --noEmit` passes.
- `npm run lint` passes.
- `npm run build` passes and `main.js` is committed.
- Manual test vault cases are listed in `test/TEST_SPEC_v2.0.0.md`.
- README explicitly states that natural-language recurrence editing in Markdown is unsupported.
- The `0.7.0` GitHub release should remain a draft until recurring timed import, Daily Note completion, and Query Block completion have been manually verified against real Todoist tasks.
